# 天翼视联点位巡检系统 - AGENTS.md

## 1. 当前阶段
- 当前阶段：Phase 0 / 首版基线与工程骨架。
- 首版唯一目标：让用户一眼看出哪些设备点位当前看不到视频画面，并立即进入派单与恢复闭环。
- 严禁发散：本阶段不做 AI 识别、大屏驾驶舱、复杂报表、SLA 自动升级、多渠道通知、多级审批。

## 2. Codex 工作方式
- 每次只处理一个小任务，优先输出 5~8 条简短计划，再开始改代码。
- 只读取与当前任务直接相关的文件，避免全仓大范围扫描。
- 不要把大段原始日志、长篇设计背景、整仓文件列表、长 diff 全量贴回线程。
- 遇到需求冲突时，不要自行扩展；优先回到本文件与基线文档核对。
- 每次提交都必须是“最小可运行增量”，不要一次铺开全部功能。

## 3. 上下文控制规则
- 根目录 `AGENTS.md` 只放稳定规则；易膨胀内容写入独立文档后按需读取。
- 不重复解释项目背景；默认以本文件为常驻背景。
- 不直接读取原始运行日志；只允许读取人工整理后的“问题结论文档”。
- 复杂任务先拆分，再逐个实现；不要在一个线程里并行推进多个大模块。
- 每个任务完成后给出“下一步最小任务”，不要一次生成后续 10 个大任务的实现。

## 4. 固定业务边界
- 业务主线固定：点位同步 -> 视频巡检 -> 异常归类 -> 故障入池 -> 通知/派单 -> 复检恢复 -> 关闭故障。
- 固定故障类型：设备离线、在线但无法播放、在线但黑屏、在线但画面异常。
- 固定工单状态：待派单、已派单、处理中、待复核、已恢复、已关闭。
- 所有故障都必须有证据：真实截图或诊断证据；拿不到截图时必须生成故障占位图。
- 点位详情页必须明确区分“平台设备信息”和“用户维护信息”。
- 坐标优先级固定：用户补录坐标 > 平台坐标 > 未定位。
- 用户可维护信息、责任映射、巡检控制、通知映射必须支持 CSV 模板导出、当前数据导出、导入预校验、导入留痕。

## 5. 首版模块优先级
### P0
运行总览、点位态势地图、视频巡检、异常处理中心、工单处理、点位详情、责任映射、企业微信通知、故障去重、恢复确认、单点重检、静默/白名单、坐标修正、CSV 导入导出、权限与留痕、主题支持。

### P1
巡检方案保存、通知手工重发、最小统计看板、故障占位图、最近成功截图回溯。

### P2
除以上之外的全部内容，统一延后。

## 6. 技术架构硬约束
- 平台：Windows 11。
- 框架：.NET 10 + WPF。
- 架构：MVVM + Generic Host + DI + 配置分层 + 结构化日志。
- 代码分层：`src/Desktop`、`src/Application`、`src/Domain`、`src/Infrastructure`、`src/Contracts`、`tests/*`。
- View 只放视图相关逻辑；业务规则必须放到 Application / Domain，不得堆进 code-behind。
- 所有平台 API 调用必须经过统一网关：签名、加解密、重试、超时、脱敏日志、异常映射全部集中处理。
- 媒体巡检、媒体预览、截图取证通过接口抽象隔离，不要把第三方媒体实现散落到 ViewModel。
- 故障去重、工单流转、通知触发必须做成可测试服务，不要写死在 UI 事件中。
- 所有外部服务都要有接口与假实现，保证离线开发、联调替换、自动化测试可行。

## 7. 地图与媒体规则
- 地图页必须使用高德地图 JS API，不得改成 WebService 坐标转换方案。
- WPF 中地图承载方式固定为 `WebView2`，由宿主与前端页面通过消息桥通信。
- 如果地图页存在覆盖层或弹出层遮挡问题，优先考虑 `WebView2CompositionControl`。
- RTSP 巡检与预览链路必须按 `rtp over tcp` 方式处理。
- 不要假设 RTSP 地址可长期复用；需要按平台返回有效期处理。

## 8. 平台接口与认证规则
- 平台 API 只允许 `HTTPS + POST + application/x-www-form-urlencoded`。
- 请求头必须带 `apiVersion=2.0`。
- 请求公共参数必须按平台协议组装：`appId`、`clientType`、`params`、`timestamp`、`version`、`signature`。
- `signature` 算法固定：`HMAC-SHA256(appId + clientType + params + timestamp + version, appSecret)`。
- `params` 固定使用 `XXTea` 加密私有参数串。
- accessToken 必须本地缓存复用，过期后刷新；禁止在每次业务请求前重新获取令牌。
- 设备在线状态必须显式区分：在线、离线、休眠（普通）、休眠（保活/AOV）。

## 9. 安全与配置规则
- 严禁把真实的 AppId、AppSecret、RSA 私钥、accessToken、refreshToken、企业微信 Webhook、地图 Key 写死进源码、示例配置或 Git 提交。
- 所有敏感配置必须从本地受保护配置读取：用户机密、环境变量、本地未提交配置文件或操作系统保护存储。
- 日志中必须对 token、密钥、签名、Webhook、手机号做脱敏。
- 示例配置文件只允许保留占位符，不允许保留真实值。

## 10. 日志规则
- 日志文件固定写在项目根目录，便于用户取用。
- 每次程序启动前清理旧日志，仅保留本轮日志。
- 必须记录：程序启动/关闭、页面关键动作、巡检任务、故障生成、工单状态流转、通知发送、CSV 导入导出、人工维护修改、API 调用明细。
- 调试与诊断信息只写日志，不在业务界面直接展示。
- Codex 不得直接消费原始日志；需要由人工或上层助手先提炼后再输入。
