## 2. 项目目标

本项目是一个基于 **.NET 10 + WPF + WebView2 + SQLite** 的 Windows 11 桌面软件，面向“天翼视联点位巡检”场景。

首版不是做炫酷展示，而是做一个能逐步闭环的最小可用系统，让用户能够逐步完成：

1. 点位同步与展示
2. 视频巡检
3. 异常归类
4. 故障入池
5. 企业微信通知/派单
6. 复检恢复
7. 关闭故障

---

## 3. 首版唯一核心目标

用户进入系统后，必须能逐步知道：

- 当前有哪些点位
- 这些点位分布在哪里
- 哪些点位当前异常
- 哪些异常已经派单
- 哪些异常已经恢复

首版的一切页面、字段、状态、日志和按钮，都必须为这条主线服务。

---

## 4. 首版固定业务主线

首版业务主线固定为：

**点位同步与展示 → 视频巡检 → 异常归类 → 故障入池 → 企业微信通知/派单 → 复检恢复 → 关闭故障**

任何新开发内容，都必须判断它是否服务于这条主线。

如果不是，就不应进入当前轮次。

## 7. 当前阶段边界

当前阶段仍然属于：

**Phase 0：点位同步与地图展示最小闭环**

当前只允许继续做与以下内容直接相关的小步收口：

- 地图异常场景兜底验证
- 地图展示细节优化
- 点位基础展示一致性
- 后续巡检链路的最小前置准备

如果要进入视频巡检，必须显式作为新阶段开始，不允许在“地图优化轮”中偷偷带入。

---

## 8. 当前推荐下一步

当前最合理的下一步不是继续铺更大的巡检闭环，而是先做一轮**单点预览异常场景人工验收**。

优先验证这 4 个场景：

1. accessToken 缺失或获取失败
2. 设备状态查询失败
3. 设备离线或休眠
4. RTSP 地址获取失败

验收目标：
- 页面不崩溃
- 提示为中文
- 日志有记录且脱敏
- 用户能明确知道当前失败原因

这轮完成后，再进入“RTSP 实际播放前的最小集成验证”。

## 10. 地图模块固定原则

地图页是首版的重要舞台，但当前阶段必须收敛。

### 10.1 已固定原则
- 地图必须使用 **高德地图 JS API**
- 宿主方式固定为 **WPF + WebView2**
- 当前地图数据来源固定为 **本地 SQLite**
- 当前不允许地图页直接查询开放平台接口

### 10.2 当前最小展示要求
- 展示已同步点位
- 支持 Marker 渲染
- 支持点击点位查看最小信息
- 显示在线 / 离线 / 未知状态
- 显示未定位点位数
- 显示未定位点位最小列表

### 10.3 当前明确不做
- 地图筛选器
- 搜索框
- 聚合点
- 海量点优化
- 轨迹
- 区域树联动
- 实时轮询平台刷新

## 11. 视频巡检阶段预告边界

后续进入视频巡检阶段时，必须仍然遵守“小步闭环”原则。

届时首个最小目标应该是：

- 选定单点或一小组点位
- 获取视频流地址
- 做最小可用巡检判断
- 明确区分：
  - 离线
  - 在线但无法播放
  - 在线但黑屏
  - 在线但画面异常
- 生成最小证据或诊断结果

在进入这个阶段之前，不得提前扩工单和通知实现。

## 12. 状态与字段设计原则

所有新增字段都必须满足以下规则：

1. 先服务当前主线，不为未来想象而扩字段
2. 先最小可用，不做超前大而全设计
3. 与现有 SQLite 表保持一致
4. 与运行总览、地图页、后续巡检页的字段口径保持一致
5. 字段命名尽量稳定，不频繁重命名

---

## 13. 技术架构固定原则

当前技术架构固定为：

- **UI**：WPF
- **桌面宿主**：.NET 10
- **模式**：MVVM
- **依赖注入与启动**：Generic Host
- **地图承载**：WebView2
- **本地存储**：SQLite
- **日志**：本地文件日志 + API 访问日志
- **开放平台接入**：统一 `OpenPlatformClient`
- **配置**：`appsettings.json` + `appsettings.Local.json`

原则：
- 不随意改框架
- 不引入重型前端工程化
- 不引入与当前阶段无关的大型依赖

---

## 14. 主题与界面原则

界面必须基于 .NET 10 主题能力与项目级统一视觉规范推进。

当前固定要求：
- 支持 `System / Light / Dark`
- 统一主色、背景层级、卡片样式、状态色
- 地图页、总览页、配置页风格保持一致
- 调试信息不直接暴露给业务用户
- 文案尽量中文统一，不中英混用

---

## 15. 日志规则

系统运行必须写日志。

### 15.1 必须记录
- 启动与关闭
- 页面关键动作
- 同步动作
- 巡检任务
- 后续故障生成
- 后续工单流转
- 后续企业微信推送
- API 调用时间、请求、响应、耗时、异常

### 15.2 当前日志原则
- 调试与诊断信息只写本地日志
- UI 只展示面向业务的结果信息
- 敏感字段必须脱敏
- 日志文件统一放在项目根目录
- 每次运行前清理旧日志，只保留当前轮结果

## 16. Codex 使用规则

后续所有 Codex 开发必须遵循以下固定规则：

1. 每一轮必须新开线程
2. 每一轮只做一个清晰范围的小任务
3. Codex 只能读取：
   - 当前提示词
   - 仓库内代码
   - 当前轮白名单文档
   - 本机运行时配置
4. 禁止读取仓库外 skill、模板、记忆文件
5. 禁止主动搜索网页，除非提示词明确允许
6. 每轮结束必须：
   - `git add`
   - `git commit`
   - `git push origin main`
7. 只允许在 `main` 分支工作
8. 不允许创建其它分支
9. 每轮审核必须同时核对：
   - Codex 完成说明
   - GitHub 远程真实代码
